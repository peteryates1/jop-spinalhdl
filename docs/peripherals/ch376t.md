# CH376T USB/SD Controller

The CH376T is a WCH chip that provides USB host and SD card access over
SPI (or parallel). On the ATARI-800-LG-V1 board it is wired in SPI mode
and connected to the FPGA via 7 signals.

---

## Hardware

### CH376T Module Pinout

| Signal | FPGA Pin | Direction | Notes |
|--------|----------|-----------|-------|
| SCK | PIN_A7 | Output | SPI clock |
| MOSI | PIN_B6 | Output | Host → CH376T data |
| MISO | PIN_B7 | Input | CH376T → host data (weak pull-up) |
| CS# | PIN_A6 | Output | SPI chip select, active low |
| INT# | PIN_B11 | Input | Interrupt, active low (weak pull-up) |
| RST | PIN_B10 | Output | Reset, **active HIGH** (high = reset) |
| SPI# | PIN_C10 | Output | Mode select, active low. Directly tied low by the design to force SPI mode. |

### Reset Sequencing

The CH376T samples the SPI# pin on the falling edge of RST (when RST
transitions from high to low). SPI# must be driven low before RST is
released. The FPGA design holds RST high for ~84 ms after configuration
using a 23-bit counter at 50 MHz, ensuring the chip enters SPI mode
reliably.

**Critical**: RST is active **HIGH**. Setting `ch376_rst = 1'b1` holds the
chip in reset. This was a source of early debugging confusion because
many chips use active-low reset.

### SPI Protocol

SPI Mode 0 (CPOL=0, CPHA=0). MSB first. Each CH376T command is
CS-framed:

1. Assert CS# (drive low)
2. Clock out the command byte
3. Wait ≥1.5 µs (post-command delay — the CH376T needs time to decode)
4. Clock out data bytes (if any) and/or clock in response bytes
5. Deassert CS# (drive high)

The post-command delay is critical. Without it, the first data byte is
corrupted. 2 µs is a safe value.

### SPI Clock Speed

The CH376T needs a slow SPI clock (~50 kHz) immediately after reset.
Once communication is verified (CHECK_EXIST succeeds), the clock can be
raised to 500 kHz or higher.

| Divider | SPI Clock | Status |
|---------|-----------|--------|
| 499 | 50 kHz | Always works, use after reset |
| 49 | 500 kHz | Works after init, good for USB |
| 24 | 1 MHz | Works |
| 4 | 5 MHz | Not tested |

Formula: `SPI_CLK = 50 MHz / (2 × (divider + 1))`

### IC Version

`GET_IC_VER` (0x01) returns 0x44 on our part. Lower 6 bits = version 4
(CH376T).

## CH376T Command Reference

### Basic Commands

| Cmd | Name | Input | Output | Description |
|-----|------|-------|--------|-------------|
| 0x01 | GET_IC_VER | — | 1 byte | IC version. Bits 5:0 = version number |
| 0x05 | RESET_ALL | — | — | Software reset. Wait 100 ms after |
| 0x06 | CHECK_EXIST | 1 byte (test) | 1 byte | Returns bitwise complement of input. Use to verify SPI communication |
| 0x0B | SET_RETRY | 2 bytes | — | Set USB retry policy (see below) |
| 0x22 | GET_STATUS | — | 1 byte | Read and clear pending interrupt status |

### SET_RETRY Format (0x0B)

Byte 1 is always 0x25 (subcommand). Byte 2 is the retry config
(per CH376DS2 §1.4):

| Bits | Meaning |
|------|---------|
| 7 | 1 = retry on NAK, 0 = report NAK as error |
| 6 | 1 + bit 7 = infinite NAK retry. 0 + bit 7 = finite retry (~3 sec) |
| 5:0 | Number of retries after USB device response timeout (0 = no retry) |

Default after reset: **0x8F** — infinite NAK retry, 15 timeout retries.

### USB Host Commands

#### Low-Level

| Cmd | Name | Input | Output | Description |
|-----|------|-------|--------|-------------|
| 0x04 | SET_USB_SPEED | 1 byte | — | Set USB bus speed. **Required for low-speed devices** |
| 0x0A | GET_DEV_RATE | 1 byte (0x07) | 1 byte | Get device speed. Bit 4: 1=low-speed, 0=full-speed |
| 0x0F | DELAY_100US | — | 1 byte | Delay 100 µs, returns non-zero when done |
| 0x13 | SET_USB_ADDR | 1 byte (addr) | — | Set target device address for subsequent transactions |
| 0x15 | SET_USB_MODE | 1 byte (mode) | 1 byte | Set operating mode. Returns 0x51 on success |
| 0x16 | TEST_CONNECT | — | interrupt | Check connection status (completes in ~2 µs) |
| 0x17 | ABORT_NAK | — | — | Abort current NAK retry |
| 0x1C | SET_ENDP6 | 1 byte | — | Set IN endpoint data toggle (see below) |
| 0x1D | SET_ENDP7 | 1 byte | — | Set OUT endpoint data toggle (see below) |
| 0x27 | RD_USB_DATA0 | — | len + data | Read USB data. Datasheet recommends for ISSUE_TKN_X and GET_DESCR |
| 0x28 | RD_USB_DATA | — | len + data | Read USB data (host mode). Both 0x27 and 0x28 work in practice |
| 0x2C | WR_HOST_DATA | len + data | — | Write data to host TX buffer. First byte = length |
| 0x41 | CLR_STALL | 1 byte (ep) | interrupt | Clear endpoint stall (CLEAR_FEATURE) |
| 0x4E | ISSUE_TKN_X | 2 bytes | interrupt | Extended token (see below) |
| 0x4F | ISSUE_TOKEN | 1 byte | interrupt | Simple token: (endpoint<<4) \| PID |

#### High-Level (CH376T handles full USB control transfer internally)

| Cmd | Name | Input | Output | Description |
|-----|------|-------|--------|-------------|
| 0x45 | SET_ADDRESS | 1 byte (addr) | interrupt | Send USB SET_ADDRESS request |
| 0x46 | GET_DESCR | 1 byte (type) | interrupt | Get descriptor (1=device, 2=config). Read data with RD_USB_DATA0 |
| 0x49 | SET_CONFIG | 1 byte (cfg) | interrupt | Send USB SET_CONFIGURATION request |
| 0x4D | AUTO_SETUP | — | interrupt | Full automatic enumeration (SET_ADDRESS + SET_CONFIG) |

### SET_ENDP6 / SET_ENDP7 Format

Per CH376DS2 §1.12–1.13, the working mode byte:

| Bits | Meaning |
|------|---------|
| 7:6 | Synchronous trigger flag: 00/01=keep, 10=set to 0 (DATA0), 11=set to 1 (DATA1) |
| 5:4 | Reserved, must be 0 |
| 3:0 | Transaction response mode, must be 0000 |

So: **0x80** = sync to DATA0, **0xC0** = sync to DATA1. Completes in ~3 µs.

### ISSUE_TKN_X Format (0x4E)

Per CH376DS2 §1.20:

| Byte | Bits | Meaning |
|------|------|---------|
| 1 (sync flag) | 7 | Synchronous trigger flag of receiver (IN toggle) |
| | 6 | Synchronous trigger flag of transmitter (OUT toggle) |
| | 5:0 | Must be 0 |
| 2 (token) | 7:4 | Destination endpoint number |
| | 3:0 | Token PID (0x0D=SETUP, 0x09=IN, 0x01=OUT) |

### USB Mode Values (SET_USB_MODE)

| Mode | Description |
|------|-------------|
| 0x00 | Disabled (default after reset) |
| 0x03 | **SD card host mode**. Manage and access files on SD card |
| 0x04 | USB host, disabled (not auto-detecting) |
| 0x05 | USB host, no SOF packets. Use for initial connection detection |
| 0x06 | USB host, automatic SOF. Use for enumeration and data transfers |
| 0x07 | USB host, SOF + bus reset. Asserts SE0 on USB D+/D- |

**Important**: USB and SD card modes are mutually exclusive. Only one
can be active at a time. Switching modes requires re-initialization
(re-enumerate USB device or re-mount SD card).

### USB Speed Values (SET_USB_SPEED)

Per CH376DS2 §1.1:

| Value | Speed | Description |
|-------|-------|-------------|
| 0x00 | 12 Mbps | Full-speed mode (default after reset) |
| 0x01 | 1.5 Mbps | Full-speed signaling at low-speed rate (non-standard) |
| 0x02 | 1.5 Mbps | **Low-speed mode. Required for most USB keyboards** |

Note: SET_USB_SPEED is reset to 12 Mbps whenever SET_USB_MODE is
called. Set the speed *after* the final SET_USB_MODE(0x06).

The device speed can be detected via GET_DEV_RATE (0x0A, input 0x07):
if bit 4 of the result is set → low-speed, otherwise full-speed. This
command is only valid in mode 0x05 (host, no SOF).

### USB PIDs (for ISSUE_TOKEN)

| PID | Value | Token Byte (EP1) |
|-----|-------|------------------|
| SETUP | 0x0D | 0x0D (EP0) |
| IN | 0x09 | 0x19 (EP1), 0x09 (EP0) |
| OUT | 0x01 | 0x01 (EP0) |

### SD Card / File System Commands

| Cmd | Name | Input | Output | Description |
|-----|------|-------|--------|-------------|
| 0x2F | SET_FILE_NAME | string + 0x00 | — | Set filename (8.3 format, "/" prefix for root). Max 14 chars incl. null |
| 0x30 | DISK_CONNECT | — | interrupt | Check disk connected. **USB storage only, not SD card** |
| 0x31 | DISK_MOUNT | — | interrupt | Initialize disk and mount FAT filesystem. Works for both USB and SD |
| 0x32 | FILE_OPEN | — | interrupt | Open file or directory. Set filename first with SET_FILE_NAME |
| 0x33 | FILE_ENUM_GO | — | interrupt | Continue directory enumeration |
| 0x36 | FILE_CLOSE | 1 byte (update) | interrupt | Close file. 0=no update, 1=update file length |
| 0x39 | BYTE_LOCATE | 4 bytes (offset) | interrupt | Seek to byte offset in file |
| 0x3A | BYTE_READ | 2 bytes (len LE) | interrupt | Read bytes from current position |
| 0x3B | BYTE_RD_GO | — | interrupt | Continue reading after buffer consumed |
| 0x3C | BYTE_WRITE | 2 bytes (len LE) | interrupt | Write bytes to current position |
| 0x3D | BYTE_WR_GO | — | interrupt | Continue writing after buffer filled |
| 0x3E | DISK_CAPACITY | — | interrupt | Get total sector count. Read 4 bytes (LE) via RD_USB_DATA0 |
| 0x3F | DISK_QUERY | — | interrupt | Get free space and FAT type. Read 12 bytes via RD_USB_DATA0 |

### File System Status Codes

| Status | Name | Meaning |
|--------|------|---------|
| 0x14 | USB_INT_SUCCESS | Operation completed successfully / end of read |
| 0x1D | USB_INT_DISK_READ | Data ready to read (call RD_USB_DATA0 then BYTE_RD_GO) |
| 0x1E | USB_INT_DISK_WRITE | Ready for write data (call WR_REQ_DATA then BYTE_WR_GO) |
| 0x1F | USB_INT_DISK_ERR | Disk operation failed |
| 0x41 | ERR_OPEN_DIR | Directory opened successfully (not a file) |
| 0x42 | ERR_MISS_FILE | File/directory not found, or end of enumeration |
| 0x43 | ERR_FOUND_NAME | Matched filename found (during enumeration) |
| 0x82 | ERR_DISK_DISCON | Disk not connected or initialization failed |
| 0x84 | ERR_LARGE_SECTOR | Sector size >512 bytes (unsupported) |
| 0x92 | ERR_TYPE_ERROR | Partition type not supported |
| 0xA1 | ERR_BPB_ERROR | Disk not formatted or parameters wrong |

### Interrupt Status Codes (USB Host)

| Status | Name | Meaning |
|--------|------|---------|
| 0x14 | USB_INT_SUCCESS | Transaction completed successfully |
| 0x15 | USB_INT_CONNECT | USB device connection detected |
| 0x16 | USB_INT_DISCONNECT | USB device disconnected |
| 0x23 | — | Data toggle mismatch (recoverable) |
| 0x28 | — | USB timeout / no device response |
| 0x2A | — | NAK (device busy, no new data) |
| 0x2B | — | Data error (recoverable) |
| 0x2C | — | USB bus error (PID error / speed mismatch) |
| 0x2E | — | STALL (endpoint halted) |

### INT# Pin Behavior

The CH376T asserts INT# (drives low) when it has a status event to
report. Reading GET_STATUS (0x22) clears the interrupt and returns the
status byte. Multiple events can queue — drain by reading GET_STATUS
repeatedly until INT# goes high.

## USB Host Keyboard: Working Initialization Sequence

This is the verified sequence that successfully enumerates a low-speed
USB keyboard. Tested on EP4CGX150 + ATARI-800-LG-V1 board with CH376T.

### Step 1: Reset and Verify

```
RESET_ALL (0x05)
wait 100 ms
CHECK_EXIST(0xA5) → expect 0x5A
```

### Step 2: Enter Host Mode (no SOF)

```
SET_USB_MODE(0x05)    → host, no SOF, returns 0x51
wait for INT# low
GET_STATUS()          → expect 0x15 (USB_INT_CONNECT)
```

The device pull-up (D- for low-speed, D+ for full-speed) triggers the
CONNECT event. No SOF packets are sent in mode 0x05, so the device
isn't disturbed.

### Step 3: Bus Reset

```
SET_USB_MODE(0x07)    → assert SE0 (USB bus reset)
wait 20 ms
SET_USB_MODE(0x06)    → release reset, start SOF packets
wait for INT# low
GET_STATUS()          → expect 0x15 (CONNECT after reset)
drain any remaining events until INT# goes high
```

### Step 4: Set USB Speed

```
SET_USB_SPEED(0x02)   → 1.5 Mbps low-speed (for keyboards)
```

**This is required for low-speed devices.** Without it, SET_ADDRESS
returns 0x2C (bus error) because the CH376T tries full-speed signaling
against a low-speed device.

### Step 5: Enumerate

```
SET_USB_ADDR(0x00)    → target address 0 (unconfigured device)
SET_ADDRESS(0x01)     → high-level: CH376T sends USB SET_ADDRESS
wait for INT#         → expect 0x14 (SUCCESS)
SET_USB_ADDR(0x01)    → update target to new address

SET_CONFIG(0x01)      → high-level: CH376T sends USB SET_CONFIGURATION
wait for INT#         → expect 0x14 (SUCCESS)
```

SET_ADDRESS (0x45) and SET_CONFIG (0x49) are high-level commands — the
CH376T handles all SETUP/DATA/STATUS token phases internally. Do NOT
try to send raw USB control transfers with ISSUE_TOKEN for these
operations; it fails with 0x2C.

### Step 6: Poll HID Keyboard

```
loop:
    SET_ENDP6(toggle ? 0xC0 : 0x80)   → set IN data toggle
    ISSUE_TOKEN(0x19)                   → IN token to endpoint 1
    wait for INT#
    if status == 0x14:                  → SUCCESS
        toggle ^= 1
        RD_USB_DATA()                   → 8-byte HID boot protocol report
    elif status == 0x2A:                → NAK, no new data
        continue
    sleep ~10 ms
```

The HID boot protocol report is 8 bytes:

| Byte | Content |
|------|---------|
| 0 | Modifier keys (bit 0=L-Ctrl, 1=L-Shift, 2=L-Alt, 3=L-GUI, 4=R-Ctrl, etc.) |
| 1 | Reserved (0x00) |
| 2-7 | Up to 6 simultaneous keycodes (0x00 = no key) |

## SD Card: Working Initialization and File Read Sequence

Tested on EP4CGX150 + ATARI-800-LG-V1 board with CH376T and 32 GB
micro SD card (FAT32).

### Step 1: Reset and Verify

```
Hardware reset (RST pin) or RESET_ALL (0x05)
wait 200 ms
CHECK_EXIST(0xA5) → expect 0x5A
```

### Step 2: Enter SD Card Mode

```
SET_USB_MODE(0x03)    → SD card host, returns 0x51
wait 100 ms
drain any pending interrupts
```

### Step 3: Mount Filesystem

```
DISK_MOUNT (0x31)
wait for INT#         → expect 0x14 (SUCCESS)
```

DISK_CONNECT (0x30) is **not needed** for SD cards — the datasheet
says it does not support SD card. DISK_MOUNT handles initialization
and mounting in one step. If it returns 0x82 (ERR_DISK_DISCON), the
SD card is not detected (check wiring).

### Step 4: Enumerate Root Directory

```
SET_FILE_NAME("/*")     → wildcard in root directory
FILE_OPEN (0x32)
loop:
    wait for INT#
    if status == 0x1D (DISK_READ):
        RD_USB_DATA0()  → 32-byte FAT directory entry
        FILE_ENUM_GO()  → continue enumeration
    elif status == 0x42 (ERR_MISS_FILE):
        break           → no more entries
```

Each directory entry is a standard 32-byte FAT structure:

| Offset | Size | Content |
|--------|------|---------|
| 0-7 | 8 | Filename (space-padded) |
| 8-10 | 3 | Extension (space-padded) |
| 11 | 1 | Attributes (0x10=dir, 0x20=archive, 0x0F=LFN) |
| 22-23 | 2 | Time (5:6:5 hour:min:sec/2) |
| 24-25 | 2 | Date (7:4:5 year-1980:month:day) |
| 26-27 | 2 | Start cluster (low word) |
| 28-31 | 4 | File size (LE) |

### Step 5: Open and Read a File

```
SET_FILE_NAME("/DIRNAME")   → first char "/" = from root
FILE_OPEN()                 → status 0x41 (ERR_OPEN_DIR) = OK
SET_FILE_NAME("FILENAME.EXT")  → no "/" = in current directory
FILE_OPEN()                 → status 0x14 (SUCCESS) = file opened

BYTE_READ(length_lo, length_hi)  → request N bytes
loop:
    wait for INT#
    if status == 0x1D (DISK_READ):
        RD_USB_DATA0()      → up to 64 bytes per chunk
        BYTE_RD_GO()        → continue reading
    elif status == 0x14 (SUCCESS):
        break               → all bytes read

FILE_CLOSE(0x00)            → close without updating length
```

For subdirectories, open each level in sequence: first the directory
(returns ERR_OPEN_DIR), then the file within it.

### Disk Capacity and Free Space

```
DISK_CAPACITY (0x3E) → SUCCESS, then RD_USB_DATA0:
  bytes 0-3: total sectors (32-bit LE)
  capacity = total_sectors × 512

DISK_QUERY (0x3F) → SUCCESS, then RD_USB_DATA0:
  bytes 0-3: total sectors (32-bit LE)
  bytes 4-7: free sectors (32-bit LE)
  byte 8: FAT type (0x01=FAT12, 0x02=FAT16, 0x03=FAT32)
```

## Gotchas and Lessons Learned

### 1. RST is Active HIGH

Unlike most chips, the CH376T uses active-high reset. Holding
`ch376_rst = 1'b1` keeps the chip permanently in reset. This caused
early bring-up failures where CHECK_EXIST returned no response.

### 2. SPI# Must Be Low Before RST Falls

The CH376T samples SPI# during the reset exit. If SPI# isn't driven low
before RST goes low, the chip enters parallel mode instead of SPI mode.
The FPGA design ties `ch376_spi_n = 1'b0` continuously and holds RST
high for ~84 ms after configuration.

### 3. Post-Command Delay is Mandatory

After clocking out the command byte, wait at least 1.5 µs before
sending data bytes. The CH376T has an internal MCU that needs time to
decode the command. Without this delay, response bytes are corrupted.

### 4. RD_USB_DATA (0x28) vs RD_USB_DATA0 (0x27)

Both commands read data from the CH376T internal buffer, but are
documented for different contexts:

- **RD_USB_DATA0 (0x27)**: Recommended by the datasheet for reading
  IN transaction data after ISSUE_TKN_X (CH376DS2 §1.20), and for
  reading descriptors after high-level GET_DESCR (CH376DS2 §1.17).
- **RD_USB_DATA (0x28)**: Documented in CH376DS1 for host-mode
  receive data.

In practice, **both work** for reading IN transaction data in host mode.
Our working keyboard code uses 0x28 after ISSUE_TOKEN (0x4F)
successfully. The FPGC6 reference code uses 0x27.

### 5. Low-Speed Keyboards Require SET_USB_SPEED

Most USB keyboards are low-speed (1.5 Mbps). The CH376T defaults to
full-speed after reset. Without calling `SET_USB_SPEED(0x02)`, all USB
transactions to a low-speed keyboard return 0x2C (bus error).

### 6. Use High-Level Enumeration Commands

The CH376T has built-in SET_ADDRESS (0x45) and SET_CONFIG (0x49)
commands that handle the complete USB control transfer internally.
Trying to do manual SETUP/DATA/STATUS token sequences with
ISSUE_TKN_X (0x4E) fails with 0x2C — the low-level token commands
don't handle the PRE preamble required for low-speed devices through
the internal hub.

### 7. Connection Can Be Flaky After Bus Reset

After SET_USB_MODE(0x07) → SET_USB_MODE(0x06), the device may report
CONNECT then DISCONNECT in quick succession. The code should retry the
entire bus reset + enumerate sequence. Keep delays short (20 ms reset
hold, 5 ms settle) — long delays increase the chance of the device
disconnecting.

### 8. Drain ALL Interrupt Events Before Enumeration

The CH376T can queue multiple status events. If a stale interrupt
(e.g., DISCONNECT from the bus reset) isn't drained before issuing
SET_ADDRESS, the wait_interrupt logic sees INT# already low and reads
the stale status instead of the SET_ADDRESS result.

Pattern: after any mode change, loop `while INT# is low: GET_STATUS()`
to drain all pending events.

### 9. SD Card DI/DO Wiring Must Be Cross-Connected

The CH376T names its SD card SPI pins from its own perspective:
- **SD_DI** (pin 6) = data IN to CH376T = MISO from SD card
- **SD_DO** (pin 18) = data OUT from CH376T = MOSI to SD card

Most micro SD breakout modules also name pins from the card's
perspective:
- **DI** = data IN to card = MOSI from host
- **DO** = data OUT from card = MISO to host

Connecting same-name to same-name (DI↔DI, DO↔DO) is **wrong** — it
connects two inputs together and two outputs together. The correct
wiring crosses them:
- CH376T SD_DO → Module DI (both are MOSI)
- CH376T SD_DI ← Module DO (both are MISO)

This was a board design error on the ATARI-800-LG-V1 prototype,
fixed with wire swap. DISK_MOUNT returns ERR_DISK_DISCON (0x82) when
the data lines are swapped.

### 10. FPGA Wait Logic Must Not Share the Reset Counter

An early bug in the FPGA design reused the hardware reset counter
(`rst_cnt`) for the INT# wait timeout. Since `ch376_rst` was derived
from `rst_active = !rst_cnt[22]`, starting a wait also asserted the
CH376T reset pin — causing the device to disconnect every time we
waited for an interrupt. Fixed by using a separate `wait_cnt` counter.

## Standalone Test Design

For clean bring-up without JOP/Atari complexity, a minimal FPGA design
(`ch376_test_top.sv`) implements a UART-to-SPI bridge. A Python script
on the host PC sends commands over a 500 kbaud serial link.

### FPGA Binary Protocol

| Command | Format | Reply | Description |
|---------|--------|-------|-------------|
| `T` | `T` + 1 byte | 1 byte (MISO) | SPI transfer one byte |
| `C` | `C` + `0`/`1` | — | CS control: 1=assert, 0=deassert |
| `D` | `D` + lo + hi | `D` | Set SPI clock divider (16-bit) |
| `P` | `P` | 0x00/0x01 | Read MISO pin |
| `I` | `I` | 0x00/0x01 | Read INT# pin (0=active) |
| `W` | `W` | `W`/`T` | Wait for INT# low (~84 ms timeout) |
| `R` | `R` | `R` | Hardware reset (~84 ms) |

### Test Scripts

| Script | Purpose |
|--------|---------|
| `tools/ch376_spi_test.py` | Basic SPI verification: CHECK_EXIST, GET_IC_VER, clock speed sweep |
| `tools/ch376_keyboard.py` | Full USB keyboard enumeration and HID polling |
| `tools/ch376_sdcard_test.py` | SD card mount, directory listing, file read |

### Build and Program

```bash
cd boards/atari800-lg-v1
/opt/altera/25.1/quartus/bin/quartus_sh --flow compile ch376_test
/opt/altera/25.1/quartus/bin/quartus_pgm -c "USB-Blaster" \
    --mode=JTAG -o "P;ch376_test.sof@1"
```

## Design Considerations

### USB and SD Card Are Mutually Exclusive

The CH376T can only operate in one mode at a time. SET_USB_MODE(0x03)
enables SD card access but disables USB host. SET_USB_MODE(0x05/0x06)
enables USB host but disables SD card. Switching requires full
re-initialization (re-enumerate USB device or re-mount filesystem).

For systems needing both (e.g., USB keyboard + SD card disk images),
options include:
1. **Two CH376T chips** — one per function, needs two SPI ports
2. **FPGA handles SD card** — use FPGA's own SPI controller for SD,
   CH376T dedicated to USB keyboard
3. **Time-slicing** — preload data from SD at boot, then switch to
   USB mode for keyboard

### Interrupt-Driven USB Keyboard Polling

With SET_RETRY default 0x8F (infinite NAK retry), issuing
ISSUE_TOKEN for an IN transaction causes the CH376T to internally
retry until data is available, then assert INT#. This enables an
interrupt-driven design:

1. Issue `ISSUE_TOKEN(0x19)` — start IN on EP1
2. Return to other work
3. INT# fires → interrupt handler reads status and data
4. Re-issue `ISSUE_TOKEN(0x19)` → back to step 2

No polling loop or sleep needed on the host side.
